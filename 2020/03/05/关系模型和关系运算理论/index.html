关系模型和关系运算理论

关系模型概述

  1. 关系模型的数据结构——关系 （二维表）

    • 关系数据结构的特点：实体和联系都用关系这种单一的数据结构来实现

  2. 关系数据结构

    • 在关系数据模型中，现实世界中的实体及实体与实体之间的联系均用关系来表示。从逻辑或用户的观点来看，关系就是二维表。

  3. 关系的操纵

    • 并
    • 交
    • 差
    • 广义笛卡尔积
    • 选择
    • 投影
    • 连接
    • 除
    • 插入
    • 删除
    • 修改
    • 关系操作的特点：操作对象和操作结果都是集合

  4. 关系是以集合的方式操作的，即操作的对象是元组的集合，操作的结果也是元组的集合。这和非关系模型的操作性结果时一条记录有着重要区别

  5. 关系的操作能力可以用代数方式和逻辑方式来表示

    • 代数方式是通过关系代数对关系的也能算来表达==查询==要求的方式、。逻辑方式是通过关系演算、用谓词表达对关系==查询==要求的方式

    • 现在关系数据库已经有了标准语言——SQL，它是一种介于关系代数和关系演算的语言

  6. 关系语言

    • 关系代数
    • 元组运算
    • 域演算
    • SQL

  7. 关系语言特点

    • 关系语言是一种高度非过程化的语言
    • 关系语言是完备的
    • 关系代数、元组关系演算和域关系演算三种语言在表达能力上完全等价
关系完整性约束

  1. 在数据库中数据完备性是指保证数据正确的特性

    • 与现实世界中应用需求的数据的相容性和正确性
    • 数据库内数据之间的相容性和正确性

  2. 在关系数据模型中一般将数据完整性分为三大类

    • 实体完整性
    • 参照完整性
    • 用户定义完整性
关系的形式化定义

  1. 关系的定义

    • 域：一组具有相同数据类型的值的集合。如整数、字符串等

    • 笛卡尔积：给定一组域D1,D1,D3……，域上的笛卡尔积为D1D2D3……={d1,d2,…dn|d1=1,2,…n}

    • (d1,d2,…dn)称为一个元组

    • di叫做元组(d1,d2,…,dn)的第i个分量

    • 关系：D1 * D2 * D3 * … * Dn的一个子集叫做域D1,D2…Dn上的关系

    • 定义在n个域上的关系称为n元关系

      域中所包含的值的个数称域的基数（用m表示）

  2. 几个概念

    • 候选码：能唯一标识元组的最小属性（组）
    • 主码：多个候选码中选定一个做主码
    • 主属性：候选码中 的诸属性
    • 非主属性：不出现在任何候选码中的属性

  3. 关系的型与值

    • 关系（表）的型：关系的结构（字段名、字段个数、域等）
    • 关系（表）的值：关系中 具体的元组，也称关系的实例

  4. 关系模式即关系的型的定义（R(U,D,dom(),F)）

    • R：关系名
    • U：属性集
    • D：关系的域
    • dom()：属性到域上的映射关系
    • F：数据依赖
    • 关系模式通常简记为：R（U）
关系的完整性约束

  1. 关系的完整性约束是对关系的正确性限定，通常是与关系所表达的实际数据约束对应

  2. 关系的完整性分为实体完整性、参照完整性和用户定义完整性

    • 实体完整性规则

      若属性A、是基本关系R的主属性，则属性A不能取空值

    • 关系模型必须遵守实体完整性规则的原因

      1. 实体完整性规则是针对基本关系而言的。一个基本表通常对应现实世界的一个实体集或多对多联系
      2. 现实世界中的实体和实体间的联系都是可区分的，即他们具有某种唯一的标识。相应的，关系模型中以主码作为唯一性标识
      3. 空值就是”不知道“或”无意义“的值。主属性取控制，就说明存在某个不可表示的实体，这与第（2）矛盾

  3. 几个概念

    设F是基本关系R的一个或一组属性但不是关系R的码。如果F与基本关系S的主码Ks相对应，则称F是基本关系R的外码。基本关系R称为参照关系，基本关系S称为被参照关系或目标关系。

    说明：

    1. 关系R和S不一定是不同的关系
    2. 目标关系S的主码KS和参照关系的外码F必须定义在同一个（或一组）域上
    3. 外码并不一定要与相应的主码同名，当外码域对应的主码属于不同关系时，往往去相同的名字，以便于识别

  4. 参照完整性规则

    若属性（若属性组）F是基本关系R的外码，它与基本关系S的主码KS相对应（基本关系R和S不一定是不同的关系），则对于R中每个元组在F上的值必须为：

    • 或者取空值（F的每个属性均为空值）
    • 或者等于S中的某个元组的主码值

  5. 用户定义完整性

    • 用户定义的完整性是针对某一具体关系数据库的约束条件，反映某一具体应用所涉及的数据必须满足的语义要求
    • 关系模型应提供定义和检验这类完整性的机制，一边用统一的系统的方法处理它们，而不要由应用程序承担这一功能。通常由RDBMS的Check约束提供这类检查
完整性约束的作用
  • 执行插入操作时检查完整性
    • 执行插入操作时需要分别检查实体完整性规则、参照完整性规则和用户定义完整性规则
  • 执行删除操作时检查完整性
    • 执行删除操作时一般只需要检查参照完整性规则
  • 执行更新操作时检查完整性
    • 执行更新操作可以看作时先删除旧的元组，然后再插入新的元组，所以执行更新操作时的完整性经检查综合了上述两种情况。
关系的数学定义
  • 关系性质
    • 每一列的数值时同类型的数据，来自同一个域（列同质）
    • 不同的列可对应同一个域，但给予不同的属性名
    • 同意关系中不允许有相同的记录。（无重复行）
    • 行的次序可以任意交换，不影响关系的实际意义。（行可调）
    • 关系中的每一个属性值都必须是不能再分的元素（分量不可分）
    • 列的次序可以任意交换，不影响关系的实际意义（列可调）
